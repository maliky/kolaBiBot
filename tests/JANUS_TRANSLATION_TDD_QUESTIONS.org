* Janus Translation TDD Questions

This file is a question worksheet for =Janus=.

=Janus= is the pure planner layer.
It receives ordered intents plus enough routed state/context to plan typed =RuntimeCommand= values.
It must not execute commands, call adapters, inspect live exchange state, or mutate strategy state.

For each question:
- proposed answer :: the default direction for tests
- counter answer :: a plausible alternative that should be rejected or defended explicitly

** 1. Boundary and contract

*** Q1.1 — What should Janus consume?

Proposed answer:
- =PairCycleState + ordered PairIntent + symbol/context=
- Explanation: Janus should plan from already-reduced state, not from raw broker events.

Counter answer:
- =EggMove + StrategyState=
- Explanation: this would pull reducer work back into Janus and blur the Isis boundary.

*** Q1.2 — What should Janus return?

Proposed answer:
- ordered =RuntimeCommand=
- Explanation: Janus is the command planner, so its output should be exchange-agnostic commands, not payload dicts and not side effects.

Counter answer:
- raw exchange payload dicts
- Explanation: this would collapse Janus into adapter-specific planning and make tests broker-shaped too early.

*** Q1.3 — Should Janus ever mutate =PairCycleState=?

Proposed answer:
- no
- Explanation: Janus should be pure planning only; state mutation belongs to Isis and upstream reducers.

Counter answer:
- yes, to stamp command ids or planning metadata
- Explanation: convenient, but it contaminates the pure planner with stateful bookkeeping.

** 2. Ordering and determinism

*** Q2.1 — Must Janus preserve intent order exactly?

Proposed answer:
- yes
- Explanation: Chronos and Isis already chose the order; Janus should not reorder without an explicit policy.

Counter answer:
- no, Janus may reorder by command priority
- Explanation: this turns Janus into a scheduler rather than a pure translator and makes tests much harder to reason about.

*** Q2.2 — If the same inputs are provided twice, must Janus return byte-for-byte equivalent commands?

Proposed answer:
- yes, modulo immutable dataclass identity
- Explanation: this is the functional-core expectation and should be frozen by tests.

Counter answer:
- no, Janus may add timestamps or generated ids
- Explanation: generated runtime ids belong outside Janus or must be injected explicitly as context.

** 3. Head planning

*** Q3.1 — How should =PLACE_HEAD= translate?

Proposed answer:
- one =RuntimeCommandKind.PLACE= command with head role and canonical order dict
- Explanation: Janus should keep the command semantic and not guess exchange-specific call names.

Counter answer:
- translate directly into exchange operation names such as =place_limit=
- Explanation: that belongs to Ogun or lower adapters.

*** Q3.2 — Should Janus ever plan =AMEND_HEAD= now?

Proposed answer:
- no unless a reducer actually emits that intent
- Explanation: do not invent head-amend behavior ahead of the reducer contract.

Counter answer:
- yes, infer amendability from state
- Explanation: inference here would bypass Isis and =step_pair()= semantics.

** 4. Tail planning

*** Q4.1 — How should =PLACE_TAIL= translate?

Proposed answer:
- one place command using the current tail-capable state and canonical tail payload
- Explanation: Janus should respect played quantity already resolved in state.

Counter answer:
- recompute tail quantity from raw strategy spec only
- Explanation: this would discard runtime state and break partial-fill planning.

*** Q4.2 — How should =AMEND_TAIL= behave when no tail identity exists yet?

Proposed answer:
- downgrade to =PLACE=
- Explanation: this preserves current intent meaning while keeping Janus deterministic.

Counter answer:
- raise an error immediately
- Explanation: stricter, but it turns a common lifecycle edge into a hard planner failure.

*** Q4.3 — How should =AMEND_TAIL= behave when one required identity is missing?

Proposed answer:
- raise a deterministic error
- Explanation: an actual amend needs both client and exchange identifiers; tests should freeze this contract.

Counter answer:
- silently emit no command
- Explanation: that hides a planner bug and makes downstream debugging harder.

** 5. Exchange agnosticism

*** Q5.1 — Should Janus know Kraken-specific ordType quirks?

Proposed answer:
- no
- Explanation: Janus should emit canonical runtime commands; broker quirks belong lower.

Counter answer:
- yes, because some command shapes are Kraken-only
- Explanation: this would leak exchange policy into the pure planner.

*** Q5.2 — Should Janus validate exchange minimum quantity or tick size?

Proposed answer:
- no
- Explanation: those are venue rules and belong at the exchange boundary or earlier validation layer.

Counter answer:
- yes, reject impossible commands early
- Explanation: attractive, but it ties Janus to exchange metadata and runtime availability.

** 6. Failure semantics

*** Q6.1 — How should Janus report unsupported intent kinds?

Proposed answer:
- raise a deterministic exception
- Explanation: unsupported intent planning is a programming error, not a runtime no-op.

Counter answer:
- emit =NOOP= or empty tuple
- Explanation: that would silently swallow coverage gaps in the reducer-planner contract.

*** Q6.2 — How should Janus behave if required state fields are absent?

Proposed answer:
- fail loudly with a stable error
- Explanation: missing state is a contract violation and should be test-visible.

Counter answer:
- guess defaults and continue
- Explanation: convenient, but dangerous for a planning layer whose main value is explicitness.

** 7. Test harvesting

*** Q7.1 — What unit tests should be the minimum Janus baseline?

Proposed answer:
- preserves intent order
- translates =PLACE_HEAD=
- translates =PLACE_TAIL=
- downgrades tail amend without identity to place
- raises on partial amend identity

Counter answer:
- start with BDD
- Explanation: Janus is mostly internal and should be frozen with unit tests first.

*** Q7.2 — Does Janus need BDD now?

Proposed answer:
- no, unless a planner behavior becomes operator-visible
- Explanation: keep Janus narrow and unit-tested.

Counter answer:
- yes, mirror all planner cases in BDD
- Explanation: this would duplicate low-level contract tests without operator value.
