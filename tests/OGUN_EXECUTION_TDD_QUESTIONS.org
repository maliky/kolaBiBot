* Ogun Execution TDD Questions

This file is a question worksheet for =Ogun=.

=Ogun= is the command executor layer.
It receives typed =RuntimeCommand= values, calls exchange-facing helpers or adapters, and returns acknowledgements or execution errors.
It must not re-plan strategy semantics and it must not reclassify lifecycle states.

For each question:
- proposed answer :: the default direction for tests
- counter answer :: a plausible alternative that should be rejected or defended explicitly

** 1. Boundary and contract

*** Q1.1 — What should Ogun consume?

Proposed answer:
- typed =RuntimeCommand= plus an exchange-facing execution context
- Explanation: Ogun executes already-planned commands; it should not receive reducer intents.

Counter answer:
- raw =PairIntent=
- Explanation: that would collapse Janus into Ogun and erase the planning seam.

*** Q1.2 — What should Ogun return?

Proposed answer:
- broker acknowledgement payload or raised execution error
- Explanation: Ogun is the first effectful layer, so exchange-facing results belong here.

Counter answer:
- updated strategy state
- Explanation: strategy-state evolution belongs upstream, not in the executor.

*** Q1.3 — Should Ogun mutate the input command?

Proposed answer:
- no
- Explanation: even if it normalizes working copies, the input command should remain observationally unchanged.

Counter answer:
- yes, pop fields directly from the command payload
- Explanation: easy in Python, but it breaks purity assumptions around reusable command values.

** 2. Dispatch semantics

*** Q2.1 — How should Ogun choose the execution function?

Proposed answer:
- by =RuntimeCommandKind= and canonical order type
- Explanation: execution dispatch should be explicit and testable.

Counter answer:
- by ad hoc payload-key heuristics
- Explanation: heuristics hide contract drift and make failures hard to trace.

*** Q2.2 — Should unsupported command kinds fail loudly?

Proposed answer:
- yes
- Explanation: unsupported execution is a contract error, not a recoverable no-op.

Counter answer:
- no, return nothing
- Explanation: silent dropping makes operator debugging much worse.

*** Q2.3 — Should unsupported order types fail loudly?

Proposed answer:
- yes
- Explanation: dispatch coverage should be explicit in tests.

Counter answer:
- coerce to nearest known type
- Explanation: coercion risks sending the wrong order to market.

** 3. Command execution cases

*** Q3.1 — How should =PLACE Market= be executed?

Proposed answer:
- call the market placement helper with normalized quantity and side
- Explanation: the executor should map command semantics to one explicit adapter call.

Counter answer:
- convert market to limit with current price
- Explanation: that changes trading semantics and does not belong in Ogun.

*** Q3.2 — How should =PLACE Limit= be executed?

Proposed answer:
- call the limit placement helper with explicit price
- Explanation: preserve command semantics and required fields.

Counter answer:
- infer price when absent
- Explanation: inference belongs earlier; execution should require a complete plan.

*** Q3.3 — How should =AMEND= be executed?

Proposed answer:
- require target order id and new planned price, then call the amend helper
- Explanation: amend must stay explicit and deterministic.

Counter answer:
- infer the order from pair name or client id alone
- Explanation: target recovery is a supervisor concern, not an executor concern.

*** Q3.4 — How should =CANCEL= be executed?

Proposed answer:
- require explicit cancelable identity and call the cancel helper
- Explanation: cancellation should not depend on hidden lookups in Ogun.

Counter answer:
- cancel all matching orders for a pair
- Explanation: that is a policy or supervisor action, not a single-command executor action.

** 4. Normalization and validation

*** Q4.1 — Should Ogun normalize numeric fields such as =Decimal= to float at the last moment?

Proposed answer:
- yes
- Explanation: this keeps typed precision upstream while satisfying adapter expectations at the edge.

Counter answer:
- no, force upstream layers to always emit adapter-ready floats
- Explanation: that leaks edge constraints back into pure layers.

*** Q4.2 — Should Ogun validate missing required fields before calling the adapter?

Proposed answer:
- yes, with stable exceptions
- Explanation: executor-side precondition checks are useful because they protect the effect boundary.

Counter answer:
- no, let the adapter crash however it does
- Explanation: that gives weaker, less localised failures.

*** Q4.3 — Should Ogun know exchange minimum quantity or tick size?

Proposed answer:
- only if the adapter contract already exposes that validation there
- Explanation: venue validation belongs at the edge if it must be enforced at execution time.

Counter answer:
- never validate anything at execution
- Explanation: pure layers should stay clean, but some edge validation may still be worth freezing here.

** 5. Error handling

*** Q5.1 — How should adapter exceptions be handled?

Proposed answer:
- bubble them up unchanged or wrap them in a narrow executor-specific error
- Explanation: tests should freeze one consistent policy.

Counter answer:
- catch everything and return =None=
- Explanation: that hides failures from Chronos and from the operator.

*** Q5.2 — Should Ogun retry failed calls automatically?

Proposed answer:
- no by default
- Explanation: retry policy is usually supervisory and time-based, not local executor policy.

Counter answer:
- yes, always retry once
- Explanation: sounds pragmatic, but it mixes execution with recovery strategy.

*** Q5.3 — Should Ogun log or persist acknowledgements itself?

Proposed answer:
- no, unless explicitly wrapped by another boundary
- Explanation: execution and persistence should stay separable for testability.

Counter answer:
- yes, store every ack directly
- Explanation: that makes Ogun harder to test and harder to reuse.

** 6. Test harvesting

*** Q6.1 — What should the minimum Ogun unit suite cover?

Proposed answer:
- place market dispatch
- place limit dispatch
- stop-family dispatch
- amend dispatch
- cancel dispatch
- missing required field failures
- unsupported order type failure

Counter answer:
- only one smoke test through the whole stack
- Explanation: that leaves too much dispatch logic unpinned.

*** Q6.2 — Does Ogun need BDD now?

Proposed answer:
- probably not
- Explanation: most Ogun behaviour is adapter-level and better frozen with unit tests or integration tests.

Counter answer:
- yes, one BDD per command type
- Explanation: that would be expensive and low-signal unless the operator surface directly exposes executor choices.
