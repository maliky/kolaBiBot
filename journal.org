* <2025-11-09 dim. 14:31> Architecture capture
  The solution splits cleanly into two coordinated applications that talk via a shared DB contract.

** 1. kolaBiTree – market data + indicators backend
   - Pulls live data from any supported platform, enriches it (indicators, metrics), and writes everything into the database.
   - Must expose a dummy generator so tests/dev can replay synthetic-yet-realistic series without exchange access.
   - Responsible for keeping derived indicators synchronized with their raw source, and for providing a documented schema/API that downstream consumers (kolaBiBot, analytics) rely on.
   - Implementation reminders:
     - Treat DB drivers via SQLAlchemy for portability (SQLite → Postgres).
     - Define versioned schemas/migrations so both real and dummy producers stay compatible.
     - Provide a narrow IPC surface (e.g., stored procs/views or service endpoints) rather than direct table poking.

** 2. kolaBiBot – market-agnostic execution layer
   - Focuses on placing, following, and managing orders across exchanges, powered by the signals exposed by kolaBiTree.
   - Reads indicators/metrics through the shared DB contract (or a thin service wrapper) and activates TSV-defined orders when conditions match.
   - Retains the existing strengths: chainable orders, parallel execution, bilingual comments, adapter-driven exchange support.
   - Near-term target: finish migrating the legacy kolaBot logic so this module is the authoritative client, while legacy files serve as regression references.

** Dual-app approach – takeaways
   - ✅ Clear separation of concerns: kolaBiTree can scale independently (async ingestion, ETL pipelines) while kolaBiBot stays lean and deterministic.
   - ✅ Testing story improves: swap kolaBiTree with its dummy twin to drive kolaBiBot end-to-end without live markets.
   - ⚠️ Requires disciplined schema/version management; breaking DB changes must be coordinated.
   - ⚠️ Latency-sensitive strategies may need a low-latency path (e.g., in-memory cache or event bus) if DB polling proves too slow.
   - Recommendation: keep two repos/modules but host shared contracts (ORM models, indicator definitions) in a small common package to avoid drift.
