   # base function
   def skew_lomax_like(z, alpL, alpR, lamL, lamR):
       y = np.zeros_like(z)
  
       left = z < 0
       right = z >= 0
  
       y[left] = (1 + (-z[left]) / lamL) ** (-(alpL + 1))
       y[right] = (1 + z[right] / lamR) ** (-(alpR + 1))
  
       return y

*** Layer 2 — Classification flags
- =New= :: confirmed/opened but not yet strategically played.
- =Failed= :: cannot continue intended pair logic.
- =Living= :: strategically played and still open.
- =Closed= :: strategically played and terminal.

*** Layer 3 — General/base order states
- =Latent= :: local order definition.
  - =Hooked= :: latent order whose hook conditions are met, ready to send.
    - =Sent= :: produced from hooked order and sent to exchange, no REST reply yet.
      - =Unadmitted= :: sent order with REST error reply, technical submission failure, not necessarily same abstraction as strategic =Failed=.
      - =Admitted= :: sent order with REST ACK reply, not yet confirmed by private DB.
        - =Confirmed= :: submitted order confirmed by private DB, then classified as =New=, =Failed=, =Living=, or =Closed=.
          - =New= :: confirmed, open, not played.
            - =Living= :: confirmed, open, played.
          - =Closed= :: confirmed, canceled, played.
          - =Failed= :: confirmed, canceled, not played, strategy failure.

*** Layer 4 — Pair-role labels

 - =Head=
   - first order in the pair.
   - hooked by public market conditions.
   - when played, it enables tail creation/hooking/submission logic.

 - =Tail=
   - second order in the pair.
   - follows the same general order lifecycle.
   - has additional relative modes because its behavior depends on the head.

*** Layer 5 — Tail-relative modes

 - =X Tail=
   - applies when the order is a head.
   - also applies when a tail is not yet associated with a played head.

 - =Flapping Tail=
   - tail associated with a =Living Head=.
   - not a base order state.
   - may be amended as partial fills appear in the private DB.

 - =Flying Tail=
   - tail associated with a =Closed Head=.
   - not a base order state.
   - sized according to the final confirmed played quantity of the head.

Goal: allow one TSV file to contain several instruments while keeping one platform per bot run.

Current state:
- =kolabi.bot run= receives one global =--exchange=, =--environment=, and =--symbol=.

- =BotService= validates one instrument and =StrategyRuntime= polls public/private state for one symbol.

Target design:
- Keep one platform per process: one =exchange=, =environment=, =market_type=, and =account_scope=.
- Add optional TSV column =symbol= after =hook=.
- If =symbol= is empty or absent, use CLI =--symbol= for backward compatibility.
- Each pair has an effective symbol before runtime starts.
- Pair name stays the logical identity and must remain unique across the whole TSV.

Pure model changes:
- Add =symbol: str | None = None= to =OrderPairSpec=.
- Keep =Chronos=, =Isis=, and =step_pair()= pure and narrow.
- Do not put exchange routing or DB lookup logic in reducers.
- Events and commands should carry the pair effective symbol at the runtime boundary.

TSV parser changes:
- Accept both old and new TSV headers.
- Old header:

- New header:
  =name tps_run essais tOut pause side oType oDelta tType tDelta atype quantity tp prix hook symbol=
- Blank =symbol= means use the run default.
- Parser tests must cover old files, new files, and blank symbol fallback.

Runtime changes:
- =BotService= materializes pair symbols before validation and runtime creation.
- Startup readiness checks public market freshness for every symbol used in the strategy.
- Private websocket readiness remains platform-level.
- Public source groups active pairs by symbol and calls =fetch_market_state(symbol)= once per symbol per poll tick.
- Market events use the pair symbol.
- Private polling keeps independent order/fill cursors per symbol so one active symbol cannot advance the cursor past another symbol's rows.
- Private record matching requires order identity and symbol.

Ogun/exchange boundary:
- Commands already carry =command.symbol=; this becomes authoritative.
- Replace the single-symbol adapter port with a one-platform symbol router.
- =SymbolRoutingExchangePort= caches one existing adapter per symbol using the same credentials, base URL, environment, and DB URLs.

- Minimum quantity checks use the row symbol.
- Tick size used by tail tracking and rounding must come from the row symbol.
- If public DB has no fresh snapshot for a TSV symbol, fail startup with a clear list of missing or stale symbols.

Operational requirement:
- Public market ingestion must already run for every symbol used in the TSV.
- This can be one poller per symbol now.
- A future multi-symbol public poller is useful but not required for this step.

- Two latent pairs on different symbols trigger two public reads.

Non-goals:

** Vocabulary definition
- Private DB :: source of truth for my orders, execution, confirmation, position, and balance.
- Public DB :: source of truth for market conditions, price/index indicators, and trigger conditions.
- Order pair :: one head and one tail.
- Latent Order ::
  - local order definition.
  - not yet submitted to the exchange.
- Hooked order ::
  - =Latent order=
  - conditions have been met based on =public db= and =private db=
  - ready to submit.
- Sent order :: an order made from the =hooked order=, sent to the exchange but no yet confirmed by =private db=.  without REST reply
- Unadmitted order :: =sent order= with REST error reply    
- Admitted order :: =sent order= with REST ack reply

- Confirmed order :: =submitted order=, with private db confirmation that =is_canceled= is either true or false.
- Played order :: =confirmed order= with a reason among, full_fill, partial_fill, post_order_failed_because_it_would_filled, stop_order_triggered, would_execute_self, would_not_reduce_position,  ioc_order_failed_because_it_would_not_be_executed
- Canceled order :: =confirmed order= with =is_cancel= true
- Opened order ::
  - =confirmed order= with =is_cancel= false
  - order which can still change either localy (eg price adjustment) or on the market plateform (eg partial fill qty) because they have not reached their final stage (closed or failed)
- New order ::  not =played= =opened order=, can be amended with my initiative
- Living order :: =played= =opened order= , can be amended with plateform initiative
- Failed order ::
  - =Canceled order= and not a =played order=
  - cannot continue the intended pair logic.
- Closed order :: =canceled order= and a =played order=

- Head ::
  - the first order of the pair.
  - created in latent state when the strategy is defined (cli args or file)
  - Hooked by public market conditions.

- Tail ::
  - The second order of the pair
  - follows the same order life cycle as any order.
- Latent Tail ::
  - local tail order definition.
  - created after the head becomes =played=.
  - not yet formally verified against tail hook conditions.
  - transient.
- Hooked Tail ::
  - =latent tail= after formal verification of hook conditions.
  - ready for automatic submission.
  - transient.
- Admitted Tail ::
  - order produced from the hooked tail.
  - sent to the platform.
  - not yet confirmed by the private DB.
- Confirmed Tail ::
  - submitted tail confirmed by the private DB.
  - then classified as =New=, =Failed=, =Living=, or =Closed= like any other order.
- Flapping Tail ::
  - a tail order associated with a =Living Head=.
  - not a separate base order state.
  - may be amended on the platform as partial fills appear in the private DB.

- Flying Tail ::
  - a =confirmed tail order= associated with a =Closed Head=.
  - not a separate base order state.
  - sized according to the final confirmed played quantity of the head.

* Databases

Demo and live databases are separated. Public and private databases are also separated.

| Purpose      | Demo DB                         | Live DB                         |
|--------------+----------------------------------+----------------------------------|
| Public book  | sqlite:///db/pub-futures-demo-PI_XBTUSD.sqlite | sqlite:///db/pub-futures-live-PI_XBTUSD.sqlite |
| Private data | sqlite:///db/prv-futures-demo.sqlite | sqlite:///db/prv-futures-live.sqlite |
| Critical private data | sqlite:///db/prv-futures-demo-critical.sqlite | sqlite:///db/prv-futures-live-critical.sqlite |

The public service stores market-book snapshots.
Public DB files are instrument-scoped. For example, =PI_ADAUSD= defaults to =sqlite:///db/pub-futures-demo-PI_ADAUSD.sqlite=.
The private service stores orders, fills, balances, positions, margin fields, and connection health.
Private DB files are account-scope scoped, not instrument-scoped. =--account-scope= is the logical account/persona label stored with private rows, for example =default= or =advers=.
Bot-side REST audit and tail telemetry use separate low-priority lanes so they cannot lock the private account DB during order/fill lifecycle decisions:

| Lane       | Default demo URL                                      | Purpose                                        |
|------------+-------------------------------------------------------+------------------------------------------------|
| Audit      | sqlite:///db/audit-futures-demo-default.sqlite        | Forensic REST send/amend/cancel audit rows     |
| Telemetry  | sqlite:///db/telemetry-futures-demo-default.sqlite    | Bot tail metrics and non-critical diagnostics  |

The runtime flags are =--audit-db-url= and =--telemetry-db-url=. These lanes are not platform truth. If they are locked or unavailable, exchange commands and lifecycle state must continue.

** Queued writer option

If SQLite lock contention persists under multi-pair stress, the next SQLite step is a per-lane queued writer: one writer thread owns one SQLAlchemy sessionmaker per DB lane, producers enqueue immutable row payloads, and the writer flushes batches in order. This keeps Chronos/Ogun non-blocking and preserves the functional boundary because reducers still emit values, not DB side effects.

** PostgreSQL persistence replacement plan

PostgreSQL is the preferred backend for long-running concurrent bots and multi-account tests. It removes SQLite's single-writer file lock and lets the current SQLAlchemy models use normal multi-connection transactions. It does not remove the need for lane separation: critical lifecycle truth, broad account state, audit, and telemetry should remain logically separated so low-priority writes cannot obscure operator reasoning.

  -e POSTGRES_DB=kolabi \

  -v kolabi-postgres-data:/var/lib/postgresql/data \
  -d postgres:16

Repository integration plan:
- Add an optional =docker-compose.postgres.yml= with the same credentials and volume.
- Add =psycopg[binary]= to the development/runtime dependency set used for SQLAlchemy PostgreSQL URLs.
- Keep the existing URL flags unchanged: =--market-db-url=, =--account-db-url=, =--critical-db-url=, =--audit-db-url=, and =--telemetry-db-url=.
- Use URL examples like =postgresql+psycopg://kolabi:kolabi@localhost:5432/kolabi= for all lanes during the first pass; later split schemas if operational clarity requires it.
- Initialise tables through the existing SQLAlchemy metadata path before running feeders. No behavioural migration is required for an empty test DB.
- Smoke test sequence: start PostgreSQL, start public feeder with a PostgreSQL =--db-url=, start private feeder with PostgreSQL =--account-db-url= and =--critical-db-url=, run =rep_xbt.tsv= with PostgreSQL audit/telemetry URLs, then compare lifecycle logs and row counts against the SQLite run.
- Acceptance: no =database is locked= errors, order/fill lifecycle still comes from the critical private lane, audit failures remain non-fatal, and existing SQLite defaults still work for local smoke tests.
