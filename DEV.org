* <2026-05-17 dim. 07:37> rewrite and vocabulary
1. A tail cannot be hooked after the failed head.
2. A tail can only be hooked after a played head.
3. Played order should not be amended.
4. Failed and Closed are terminal states
5. Tail is creation :
   - Latent -- Hooked -- submitted tail when head is Played     
   - Flapping when head is Living
   - Flying when head is Closed
6. In case of Flapping Tail we only have one tail, that is amended on the plateform as partial_fills comes in the private db.
7. If a head is partially filled and then canceled, then it should live behind flying tail associated with the partialy filled amound.
8. When an order is hooked the next automatic step is the submission.
   
** Vocabulary definition
- Private DB :: source of truth for my orders, execution, confirmation, position, and balance.
- Public DB :: source of truth for market conditions, price/index indicators, and trigger conditions.
- Order pair :: one head and one tail.
- Latent Order ::
  - local order definition.
  - not yet submitted to the exchange.
- Hooked order ::
  - =Latent order=
  - conditions have been met based on =public db= and =private db=
  - ready to submit.
- Submitted order :: an order made from the =hooked order=, sent to the exchange but no yet confirmed by =private db=.
- Confirmed order :: =submitted order=, with private db confirmation that =is_canceled= is either true or false.
- Played order :: =confirmed order= with a reason among, full_fill, partial_fill, post_order_failed_because_it_would_filled, stop_order_triggered, would_execute_self, would_not_reduce_position,  ioc_order_failed_because_it_would_not_be_executed
- Canceled order :: =confirmed order= with =is_cancel= true
- Opened order ::
  - =confirmed order= with =is_cancel= false
  - order which can still change either localy (eg price adjustment) or on the market plateform (eg partial fill qty) because they hqve not reached their final stage (closed or failed)
- New order ::  not =played= =opened order=, amended with my initiative
- Living order :: =played= =opened order= , amended with plateform initiative
- Failed order ::
  - =Canceled order= and not a =played order=
  - cannot continue the intended pair logic.
- Closed order :: =canceled order= and a =played order=

- Head ::
  - the first order of the pair.
  - created in latent state when the strategy is defined (cli args or file)
  - Hooked by public market conditions.
    
- Tail ::
  - The second order of the pair
  - follows the same order life cycle as any order.
- Latent Tail ::
  - local tail order definition.
  - created after the head becomes =played=.
  - not yet formally verified against tail hook conditions.
  - transient.
- Hooked Tail ::
  - =latent tail= after formal verification of hook conditions.
  - ready for automatic submission.
  - transient.
- Submitted Tail ::
  - order produced from the hooked tail.
  - sent to the platform.
  - not yet confirmed by the private DB.
- Confirmed Tail ::
  - submitted tail confirmed by the private DB.
  - then classified as =New=, =Failed=, =Living=, or =Closed= like any other order.    
- Flapping Tail ::
  - a tail order associated with a =Living Head=.
  - not a separate base order state.    
  - may be amended on the platform as partial fills appear in the private DB.

- Flying Tail ::
  - a =confirmed tail order= associated with a =Closed Head=.
  - not a separate base order state.    
  - sized according to the final confirmed played quantity of the head.

* Databases

Demo and live databases are separated. Public and private databases are also separated.

| Purpose      | Demo DB                         | Live DB                         |
|--------------+----------------------------------+----------------------------------|
| Public book  | sqlite:///pub-futures-demo.sqlite | sqlite:///pub-futures-live.sqlite |
| Private data | sqlite:///prv-futures-demo.sqlite | sqlite:///prv-futures-live.sqlite |

The public service stores market-book snapshots.
The private service stores orders, fills, balances, positions, margin fields, and connection health.

Private account state is the source of truth for our own orders.

Instrument rule cache:

- Kraken instrument metadata is cached locally in the public DB.
- Demo location:
  - DB file: =pub-futures-demo.sqlite=
  - table: =exchange_instruments=
- Live location:
  - DB file: =pub-futures-live.sqlite=
  - table: =exchange_instruments=
- Cached fields include:
  - =symbol=
  - =tick_size=
  - =contract_size=
  - =min_quantity=
  - =instrument_type=
  - =tradeable=
  - =raw_payload=
