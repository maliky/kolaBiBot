*** Layer 2 — Classification flags
- =New= :: confirmed/opened but not yet strategically played.
- =Failed= :: cannot continue intended pair logic.
- =Living= :: strategically played and still open.
- =Closed= :: strategically played and terminal.

*** Layer 3 — General/base order states
- =Latent= :: local order definition.
  - =Hooked= :: latent order whose hook conditions are met, ready to send.
    - =Sent= :: produced from hooked order and sent to exchange, no REST reply yet.
      - =Unadmitted= :: sent order with REST error reply, technical submission failure, not necessarily same abstraction as strategic =Failed=.
      - =Admitted= :: sent order with REST ACK reply, not yet confirmed by private DB.
        - =Confirmed= :: submitted order confirmed by private DB, then classified as =New=, =Failed=, =Living=, or =Closed=.
          - =New= :: confirmed, open, not played.
            - =Living= :: confirmed, open, played.
          - =Closed= :: confirmed, canceled, played.
          - =Failed= :: confirmed, canceled, not played, strategy failure.

*** Layer 4 — Pair-role labels

 - =Head=
   - first order in the pair.
   - hooked by public market conditions.
   - when played, it enables tail creation/hooking/submission logic.

 - =Tail=
   - second order in the pair.
   - follows the same general order lifecycle.
   - has additional relative modes because its behavior depends on the head.

*** Layer 5 — Tail-relative modes

 - =X Tail=
   - applies when the order is a head.
   - also applies when a tail is not yet associated with a played head.

 - =Flapping Tail=
   - tail associated with a =Living Head=.
   - not a base order state.
   - may be amended as partial fills appear in the private DB.

 - =Flying Tail=
   - tail associated with a =Closed Head=.
   - not a base order state.
   - sized according to the final confirmed played quantity of the head.

** Vocabulary definition
- Private DB :: source of truth for my orders, execution, confirmation, position, and balance.
- Public DB :: source of truth for market conditions, price/index indicators, and trigger conditions.
- Order pair :: one head and one tail.
- Latent Order ::
  - local order definition.
  - not yet submitted to the exchange.
- Hooked order ::
  - =Latent order=
  - conditions have been met based on =public db= and =private db=
  - ready to submit.
- Sent order :: an order made from the =hooked order=, sent to the exchange but no yet confirmed by =private db=.  without REST reply
- Unadmitted order :: =sent order= with REST error reply    
- Admitted order :: =sent order= with REST ack reply

- Confirmed order :: =submitted order=, with private db confirmation that =is_canceled= is either true or false.
- Played order :: =confirmed order= with a reason among, full_fill, partial_fill, post_order_failed_because_it_would_filled, stop_order_triggered, would_execute_self, would_not_reduce_position,  ioc_order_failed_because_it_would_not_be_executed
- Canceled order :: =confirmed order= with =is_cancel= true
- Opened order ::
  - =confirmed order= with =is_cancel= false
  - order which can still change either localy (eg price adjustment) or on the market plateform (eg partial fill qty) because they have not reached their final stage (closed or failed)
- New order ::  not =played= =opened order=, can be amended with my initiative
- Living order :: =played= =opened order= , can be amended with plateform initiative
- Failed order ::
  - =Canceled order= and not a =played order=
  - cannot continue the intended pair logic.
- Closed order :: =canceled order= and a =played order=

- Head ::
  - the first order of the pair.
  - created in latent state when the strategy is defined (cli args or file)
  - Hooked by public market conditions.

- Tail ::
  - The second order of the pair
  - follows the same order life cycle as any order.
- Latent Tail ::
  - local tail order definition.
  - created after the head becomes =played=.
  - not yet formally verified against tail hook conditions.
  - transient.
- Hooked Tail ::
  - =latent tail= after formal verification of hook conditions.
  - ready for automatic submission.
  - transient.
- Admitted Tail ::
  - order produced from the hooked tail.
  - sent to the platform.
  - not yet confirmed by the private DB.
- Confirmed Tail ::
  - submitted tail confirmed by the private DB.
  - then classified as =New=, =Failed=, =Living=, or =Closed= like any other order.
- Flapping Tail ::
  - a tail order associated with a =Living Head=.
  - not a separate base order state.
  - may be amended on the platform as partial fills appear in the private DB.

- Flying Tail ::
  - a =confirmed tail order= associated with a =Closed Head=.
  - not a separate base order state.
  - sized according to the final confirmed played quantity of the head.

* Tail Fix Trace 2026-05-25

I changed live runtime flow so a head REST ack only emits HEAD_SUBMITTED; HEAD_PLAYED now comes from private order polling, which gates tail placement on confirmed fill. I added raw exchange interaction logging at the adapter port boundary to `logs/kraken-exchange-raw.ndjson` with request, ack, and raw payload fields for deterministic troubleshooting. I also kept trigger-order verification and updated runtime tests to require private fill confirmation before tail submission.

Limitations: if private feed is stale or unavailable, tail placement is delayed or can time out by design; this fails loudly rather than placing a potentially invalid reduce-only trigger too early.

* Tail Amend Ground-Truth Fix 2026-05-27

The observed non-updating tail was not only a pure reducer problem: REST audit showed repeated =/editorder= replies with =editStatus.status=invalidPrice= for unrounded stop amendments such as =74757.668749999999998470=, while placement was rounded correctly. I fixed this at the Kraken adapter boundary by rounding amend =price= and =stopPx= to the instrument tick before sending =/editorder=.

There was also a semantic split: local telemetry treated =-y S= as =src=book= while Kraken placement defaulted the stop trigger to last price. I made =S= and =S-= resolve to LastPrice, with =Sf/Sf-= for MarkPrice and =Si/Si-= for IndexPrice, so telemetry, order building, and exchange trigger mapping use the same source.

State is now grounded more strictly in exchange acknowledgement: tail amend accepts update the confirmed stop from the adapter ack, rejected amend acks revert the desired stop back to the last confirmed stop. Telemetry can still show market/reference movement, but =tail_stop= should only move when the platform has accepted or confirmed the new stop.

* Chronos Repeat and Logging Review 2026-05-27

** Findings

- Active core path is now concentrated in =StrategyRuntime -> Chronos -> Isis -> step_pair -> Horus -> Ogun=. The old =kolabi/runtime/kola= and archived runners are still present for reference, but they are not on the active =kolabi.bot= path.
- The main live risk was not inside the pure reducer. =StrategyRuntime.run()= awaited exchange execution inline, so a REST call or tail trigger verification could stop Chronos from draining queued private/public events. That can delay =closed head -> tail submitted= visibility and can also delay repeat scheduling.
- Pair repetition existed in the vocabulary (=try_num= and =dr_pause=) but had no active Chronos scheduler. A terminal pair could finish cleanly and the runtime would stop instead of re-hooking the next attempt.
- Chronos command deduplication was too coarse for repetition. It keyed a command by pair and command shape before =StrategyRuntime= generated a fresh client order id, so a second =PLACE_HEAD= for the same pair could be treated like an old command.
- Logging was close but not precise enough for operator review: =CS= could show the unrounded desired stop from the request instead of the stop price returned by the exchange acknowledgement. The =# >= note in =testRepB.log= was correct.
- Test overlap remains: =tests/bot/test_chronos_supervisor.py= and =tests/bdd/test_chronos_supervisor_bdd.py= cover the same Chronos behaviours. Keep both for now: the classic test is the fast precise regression surface; the BDD file is the operator-readable behaviour contract. If this becomes noisy, make BDD canonical and reduce the classic file to edge cases only.

** Changes Implemented

- Added Chronos-owned repeat scheduling with per-pair =attempt_index=. Terminal pairs repeat until =try_num= is exhausted, honour =dr_pause= through =pending_repeats=, and re-hook the next attempt through the same pure reducer path.
- Changed Chronos command dedupe to include the active attempt number, so a repeated pair can emit a fresh =PLACE_HEAD= without bypassing duplicate protection inside one attempt.
- Moved command execution out of the main runtime event loop into background command tasks. Chronos can keep processing queued events and ready repeats while Ogun/exchange execution is in flight.
- Kept blocking exchange calls at the adapter boundary configurable: active BotService ports run blocking exchange calls in a thread, while unit tests keep direct calls to avoid test-runner thread hangs in the sandbox.
- Tail submitted follow-up events now prefer =ack.price= over the raw requested =stopPx=, so the confirmed stop and =CS= display the tick-rounded exchange acknowledgement.
- Runtime logs now include attempt index with =UPDATE (pair#attempt)= and =METRICS (pair#attempt)=, and =CS= means confirmed/platform-grounded stop while =DS= remains the local desired stop.

** Remaining Limits

- I did not delete legacy tests or archived runtime code in this pass. The review found redundancy, but not a safe deletion target that is clearly unrelated to the active path without a broader cleanup commit.
- The active runtime still has transitional names such as =OrderState= shared by head and tail. That is not blocking this fix, but it remains part of the typed-cleanup backlog.

* Reinforced Runtime Plan Implemented 2026-05-28

** Clarified Terms

- Terminal pair means one pair attempt whose lifecycle cannot progress further in this attempt: =head_state=failed=, or =head_state=closed= with either no played qty or tail terminal (=tail_state=closed|failed=).
- In-flight command means one exchange command already sent to Ogun/adapter and still waiting for ack/exception. In-flight tracking is now keyed by =pair_name + attempt_index + role=.
- Reducer means the pure state transition function (=step_pair= via Isis). It can look non deterministic if tick rounding only happens at adapter boundary; now tick size is carried into market tick events and used in pure tail update decisions, so reducer output is stable for the same event stream.

** Why Chronos Had Internal Queues Before

- It was an intermediate shell pattern from earlier async experiments where Chronos owned event batching directly.
- After StrategyRuntime became the single async owner, keeping a second queue path in Chronos only duplicated responsibilities and complicated tests, so it has been removed.

** What Was Implemented

- Removed Chronos internal queue API (=event_queue=, =command_queue=, =run_once()= batch drain path). StrategyRuntime is now the only async queue owner.
- Added strict head-fill anchor for relative tails: private polling only sets tail reference from private fill execution price. Public price fallback was removed for this path.
- Added 20s grace waiting for missing private fill reference on relative tails. If no fill price appears within grace, runtime fails loudly with pair/order identities, instead of placing a misanchored tail.
- Added per-slot command serialisation in runtime: slot key is =pair + attempt + role=. Commands in the same slot are serialised; different slots run concurrently.
- Added tail amend coalescing: while one tail amend is in flight for a slot, only the latest pending amend is kept.
- Wired instrument tick size from runtime market state into public market tick events and into =step_tail_trail(..., tick_size=...)=.
- Updated client order id format to attempt-aware readable ids: =H<attempt><word>-<yymmddHHMMSS>= and =T<attempt><word>-<yymmddHHMMSS>=.

** Failure Cases For 20s Fill Grace

- Private fills stream delayed or stale while order stream is fresh.
- Exchange emits order terminal status before the corresponding fill row is persisted locally.
- Fill row exists but has missing/invalid execution price in payload.
- DB cursor lag or temporary connectivity issue prevents fill row read in time.

  -> StrategyRuntime slot dispatcher
      slot free: execute now (Ogun -> Adapter -> Exchange)
      slot busy: queue, and for tail amend keep latest only
  -> ack/failure followup event -> StrategyRuntime queue
  -> Chronos state update, repeat scheduling, and next dispatch
