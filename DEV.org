In order to have market makers, even in a simple state, we need a way to adjust orders to their fill frequency.

We work with pairs, and a failed market-making attempt is when the tail is filled without having been updated. In that case, we should have a way to reduce the frequency of that failed order type.

The idea is to adjust the pair =tOut=. If a pair fails, then =tOut= and the pause are adjusted.
For example, if =tOut= is 1 minute and the pause is 1 minute, and the pair fails after waiting x times, then the pause can be adjusted by d with d an asymetric  function of x (ressembly the s shape curve of daniel kaneham in thinking fast and slow) with a range of roughly -1 1 and a x in 0 +. Given our slow bot, the minimum =tOut= should probably be around 30 seconds, so the maximum pause should be 1 minute 30 seconds. Between those bounds, we can have any =tOut= and complementary pause length (kind of liptchisienne)

This adjustment the time delta between the order reaching the market and the time it is filled (failed fill). In that case, it should be an inverse function, meaning that the shorter the time to fill, the longer we should extend the pause.

The adjustment function should have a 0 to 1 domain and a 0 to 1 range, with an S shape similar to the one we use to shrink the tail distance. To make it direct rather than inverse, say that the longer we wait for the fill, with a maximum of 1 minute, the less we remove from =tOut=. In fact, we can centre the S curve at 30 seconds. If we wait less than 30 seconds, =tOut= is reduced. More precisely, if we wait less than half of the run time, we shrink the run time. If we wait more than half of the run time, we increase the run time, with a maximum of 1.5 minutes and a minimum of 30 seconds.

As you can see, this is done independently from the other side, and is autonomous for one pair.

It could be implemented by changing the =tOut= strategy parameter, making it a range, in our case 0.5 to 1.5, plus a standard adjustment function. The check to make is that the pause, or possibly the default pause, should match the amplitude of the configured timeout.

** Runtime log keywords
These are runtime/operator log labels, not exchange feed labels. Each entry says when the keyword appears and what it means for the operator.

- Arms a latent timeout :: (LATENT_TIMEOUT_ARMED) Appears when a timeout is started for a latent head that is waiting on a trigger or gate.
- Blocks an unsafe order action :: (ORDER_SAFETY_BLOCKED) Appears when the runtime or admin cleanup cannot safely act, usually because identity or ownership evidence is missing.
- Closes a known live order lease :: (LEASE_CLOSED) Appears when private exchange evidence marks a tracked lease closed, cancelled, or filled.
- Confirms a head closure :: (HEAD_CANCELLED) Appears on the private confirmation that the head closed or cancelled; played quantity is recorded for tail decisions.
- Confirms an orphan closed :: (ORPHAN_CLOSED) Appears when startup cleanup observes that an orphan order is no longer open.
- Confirms tail visibility :: (TAIL_VISIBLE) Appears after a previously pending tail becomes visible in private exchange state.
- Evaluates a price gate :: (GATE_WAIT-2) Appears when a latent head is checked against the market price; it can mean the gate is ready, still waiting, outside the trigger, or missing a baseline price.
- Finds an orphan order :: (ORPHAN_FOUND) Appears at service startup when an exchange order still looks open but is not owned by the fresh runtime state.
- Ignores a stale head cancel acknowledgement :: (HEAD_CANCEL_ACK_STALE) Appears when an ACK belongs to an older head attempt and must not affect the current attempt.
- Introduces the field order for a runtime label :: (LEGEND--KEYWORD) Appears once near runtime startup; it gives the field names used by later lines with the same keyword.
- Makes a repeat ready :: (REPEAT_READY) Appears when the repeat pause has elapsed and the repeated pair is back at its gate/launch step.
- Marks a lease as awaiting cancel confirmation :: (CANCEL_PENDING) Appears after the runtime sends a cancel against a tracked lease and waits for private closure evidence.
- Receives head admission :: (HEAD_ACK) Appears after the exchange adapter acknowledges the head placement; the runtime can then arm fill and visibility timers.
- Records a state change :: (UPDATE) Appears when a pair, side, status, quantity, or stop value changes; it is the broad lifecycle trace.
- Rejects a tail amend :: (AMEND_REJECTED) Appears when the exchange or adapter rejects the amend; the runtime should keep or restore the confirmed stop value.
- Releases a chained pair :: (CHAIN_READY) Appears when an upstream close or fill satisfies a dependency and a hooked downstream pair can become active.
- Reports a command execution failure :: (COMMAND_FAILED) Appears when a runtime command raised inside the executor; it flags exchange or adapter failure around the attempted action.
- Reports a gate check error :: (GATE_WAIT-ERR) Appears when runtime cannot fetch or interpret market state for a gate check; the head remains latent.
- Reports tail telemetry :: (METRICS) Appears while a tail is living, submitted, or hooked; it records values used to judge trailing and amend behaviour.
- Retries a cancel :: (CANCEL_RETRY) Appears when a lease stayed open beyond the cancel retry window; the runtime sends another cancel request.
- Schedules a repeat :: (REPEAT_WAIT) Appears after a completed pair is scheduled to restart after its pause.
- Sends a cancel command :: (CANCEL_SENT) Appears when a cancel is sent for a reason other than the direct head-timeout path.
- Sends a head order :: (HEAD_SENT) Appears when the runtime dispatches the head placement command to the exchange adapter.
- Sends a head timeout cancel :: (HEAD_CANCEL_SENT) Appears when the runtime sends the cancel caused by =HEAD_TIMEOUT=.
- Sends a tail amend :: (AMEND_SENT) Appears when the runtime sends a stop/price amend for a live tail; it records the current and desired values.
- Sends an orphan cancel :: (ORPHAN_CANCEL_SENT) Appears when startup cleanup sends a cancel for an orphan order.
- Starts a repeat :: (REPEAT_START) Appears when the repeated pair emits its next commands.
- Suppresses a replace-cancel side effect :: (AMEND_REPLACE_CANCEL_SUPPRESSED) Appears during cancel-replace amend handling when the cancel half should not be treated as a real tail close.
- Times out a latent head :: (LATENT_TIMEOUT) Appears when a latent head waited too long before becoming placeable; the runtime abandons or advances according to the pair rules.
- Times out a live head :: (HEAD_TIMEOUT) Appears when a head exceeded its configured =tOut= window; the next expected action is cancellation.
- Tracks one known live order lease :: (LEASE_OPEN) Appears when the runtime opens or observes a live head or tail order identity; it means the runtime is responsible for tracking and eventually closing that lease.
- Waits for head capacity :: (HEAD_WAIT) Appears when a head is otherwise ready but the maximum number of active heads is already reached.
- Waits for tail amend confirmation :: (AMEND_PENDING) Appears when an amend did not confirm by the configured deadline.
- Waits for tail visibility :: (TAIL_PENDING) Appears when a tail placement was sent but private visibility did not confirm the tail by the deadline.
- Waits on a latent gate prerequisite :: (GATE_WAIT-1) Appears while a latent head is still blocked by its hook dependency or start-time window.
- Warns that head visibility is pending :: (HEAD_VISIBILITY_PENDING) Appears when placement was acknowledged but private visibility did not arrive before the visibility timeout.

*** Layer 2 — Classification flags
- =New= :: confirmed/opened but not yet strategically played.
- =Failed= :: cannot continue intended pair logic.
- =Living= :: strategically played and still open.
- =Closed= :: strategically played and terminal.

*** Layer 3 — General/base order states
- =Latent= :: local order definition.
  - =Hooked= :: latent order whose hook conditions are met, ready to send.
    - =Sent= :: produced from hooked order and sent to exchange, no REST reply yet.
      - =Unadmitted= :: sent order with REST error reply, technical submission failure, not necessarily same abstraction as strategic =Failed=.
      - =Admitted= :: sent order with REST ACK reply, not yet confirmed by private DB.
        - =Confirmed= :: submitted order confirmed by private DB, then classified as =New=, =Failed=, =Living=, or =Closed=.
          - =New= :: confirmed, open, not played.
            - =Living= :: confirmed, open, played.
          - =Closed= :: confirmed, canceled, played.
          - =Failed= :: confirmed, canceled, not played, strategy failure.

*** Layer 4 — Pair-role labels

 - =Head=
   - first order in the pair.
   - hooked by public market conditions.
   - when played, it enables tail creation/hooking/submission logic.

 - =Tail=
   - second order in the pair.
   - follows the same general order lifecycle.
   - has additional relative modes because its behavior depends on the head.

*** Layer 5 — Tail-relative modes

 - =X Tail=
   - applies when the order is a head.
   - also applies when a tail is not yet associated with a played head.

 - =Flapping Tail=
   - tail associated with a =Living Head=.
   - not a base order state.
   - may be amended as partial fills appear in the private DB.

 - =Flying Tail=
   - tail associated with a =Closed Head=.
   - not a base order state.
   - sized according to the final confirmed played quantity of the head.

- New order ::  not =played= =opened order=, can be amended with my initiative
- Living order :: =played= =opened order= , can be amended with plateform initiative
- Failed order ::
  - =Canceled order= and not a =played order=
  - cannot continue the intended pair logic.
- Closed order :: =canceled order= and a =played order=

  - the first order of the pair.
  - created in latent state when the strategy is defined (cli args or file)
  - Hooked by public market conditions.

- Tail ::
  - The second order of the pair
  - follows the same order life cycle as any order.

  - local tail order definition.

  - =latent tail= after formal verification of hook conditions.

#+BEGIN_SRC bash  -i
#!/usr/bin/env bash

# Set up authentication:
API_KEY="put your own API Key here"
PRIVATE_KEY_PATH="test-prv-key.pem"

# Set up the request:
API_METHOD="POST"
API_CALL="api/v3/order"
API_PARAMS="symbol=BTCUSDT&side=SELL&type=LIMIT&timeInForce=GTC&quantity=1&price=0.2"

# Sign the request:
timestamp=$(date +%s000)
api_params_with_timestamp="$API_PARAMS&timestamp=$timestamp"
signature=$(echo -n "$api_params_with_timestamp" \
            | openssl dgst -sha256 -sign "$PRIVATE_KEY_PATH" \
            | openssl enc -base64 -A)

# Send the request:
curl -H "X-MBX-APIKEY: $API_KEY" -X "$API_METHOD" \
    "https://testnet.binance.vision/$API_CALL?$api_params_with_timestamp" \
    --data-urlencode "signature=$signature"

* Databases

Demo and live databases are separated. Public and private databases are also separated.

| Purpose      | Demo DB                         | Live DB                         |
|--------------+----------------------------------+----------------------------------|
| Public book  | postgresql+psycopg://kolabi:kolabi@127.0.0.1:15433/kolabi_market | postgresql+psycopg://kolabi:kolabi@127.0.0.1:15433/kolabi_market |
| Private data | postgresql+psycopg://kolabi:kolabi@127.0.0.1:15433/kolabi_account | postgresql+psycopg://kolabi:kolabi@127.0.0.1:15433/kolabi_account |
| Critical private data | postgresql+psycopg://kolabi:kolabi@127.0.0.1:15433/kolabi_critical | postgresql+psycopg://kolabi:kolabi@127.0.0.1:15433/kolabi_critical |

The public service stores market-book snapshots.
All instruments share the PostgreSQL public market lane. Public rows remain scoped by exchange, environment, market type, and symbol.
The private service stores orders, fills, balances, positions, margin fields, and connection health.
Private DB lanes are account-scope scoped, not instrument-scoped. =--account-scope= is the logical account/persona label stored with private rows, for example =default= or =advers=.
Bot-side REST audit and tail telemetry use separate low-priority lanes so they cannot lock the private account DB during order/fill lifecycle decisions:

| Lane       | Default demo URL                                      | Purpose                                        |
|------------+-------------------------------------------------------+------------------------------------------------|
| Audit      | postgresql+psycopg://kolabi:kolabi@127.0.0.1:15433/kolabi_audit       | Forensic REST send/amend/cancel audit rows     |
| Telemetry  | postgresql+psycopg://kolabi:kolabi@127.0.0.1:15433/kolabi_telemetry   | Bot tail metrics and non-critical diagnostics  |

The runtime flags are =--audit-db-url= and =--telemetry-db-url=. These lanes are not platform truth. If they are locked or unavailable, exchange commands and lifecycle state must continue.

Critical order/fill forensic shielding is enabled by default. Private-feed maintenance must not prune raw =private_ws= or =private_ws_critical= payloads, nor their ingest audit rows, unless the operator passes =--allow-critical-forensic-prune=. Account state, REST audit, and tail telemetry stay bounded by their own retention knobs because they are diagnostic or sampled lanes, not lifecycle ground truth.

** PostgreSQL persistence

PostgreSQL is the =kolabidb= backend for long-running concurrent bots and multi-account tests. It lets the current SQLAlchemy models use normal multi-connection transactions. It does not remove the need for lane separation: critical lifecycle truth, broad account state, audit, and telemetry should remain logically separated so low-priority writes cannot obscure operator reasoning.

Host findings:
- This host already has =postgres:16= locally.
- Existing Tusis development PostgreSQL exposes =127.0.0.1:15432->5432=, so Kolabi uses =15433= by default.
- Existing Tusis containers use =PGDATA=/var/lib/postgresql/data/pgdata= and =pg_isready= for health.

Kolabi Docker files:
- =docker-compose.postgres.yml= defines service =kolabi-db= with =postgres:16=, host port =127.0.0.1:${KOLABI_POSTGRES_PORT:-15433}=, and a bind mount at =./dbs/postgres:/var/lib/postgresql/data=.
- =docker/postgres/init/001-create-databases.sql= creates separate lane databases for the main bot and the adversarial bot/account.
- =psycopg[binary]= is a runtime dependency because SQLAlchemy PostgreSQL URLs use the =postgresql+psycopg= driver.

Pros of storing PostgreSQL files under =dbs/postgres=:
- The DB location is visible under the repo-local =dbs= directory.
- Backups, local resets, and disk-usage checks are straightforward.
- The repo-local helper can manage all DB lanes with one mental model.

Cons:
- Docker no longer owns the storage path as an opaque named volume.
- File ownership may appear as the Postgres container UID on the host.
- Accidental manual deletion is easier, so =dbs/postgres/= must stay ignored by Git.

Databases created on first volume initialisation:
| Lane | Main bot DB | Adversarial bot DB |
|------+-------------+--------------------|
| Public market | kolabi_market | shared kolabi_market |
| Private account | kolabi_account | kolabi_account_advers |
| Critical private | kolabi_critical | kolabi_critical_advers |
| REST audit | kolabi_audit | kolabi_audit_advers |
| Bot telemetry | kolabi_telemetry | kolabi_telemetry_advers |

Acceptance:
- No persistence backpressure warnings in the runtime logs.
- Main and adversarial private lanes remain separated.
- Both bots share the public market lane.
- Order/fill lifecycle still comes from the critical private lane.
- Audit and telemetry writes remain non-critical.

Queued writer note: the queued writer remains compatible with PostgreSQL, but is deferred until PostgreSQL testing shows whether audit/telemetry still need buffering. If implemented later, keep it as an edge adapter only; reducers, Isis, Chronos, and pair semantics must not know about queues or writer threads.
